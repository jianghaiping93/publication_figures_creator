# Repository Guidelines

## Project Structure & Module Organization
- `docs/github_ref.md` stores external reference links for figure resources.
- There is no application source tree in this repository; content is documentation-only.
- Add new reference material under `docs/` using clear, topical filenames (example: `docs/diagram_refs.md`).

## Build, Test, and Development Commands
- There are no build, test, or runtime commands defined in this repository.
- If you add scripts or tooling later, document them here with exact commands (example: `make lint`, `python -m pytest`).

## Coding Style & Naming Conventions
- Use Markdown for documentation.
- Keep lines reasonably short (around 100 characters) for readability.
- Name files with lowercase, underscore-separated names (example: `figure_sources.md`).
- Use clear, descriptive headings in docs and avoid duplicate section titles.

## Testing Guidelines
- No test framework is configured.
- If tests are introduced, document the framework, test location, naming pattern (example: `test_*.py`), and how to run them.

## Commit & Pull Request Guidelines
- Git history is not available in this workspace, so no existing commit conventions can be verified.
- Suggested commit format: short, imperative summary (example: `Add references for bio figure tools`).
- PRs should include a concise description of what changed and why.
- PRs should include links to relevant sources if you add or update references.
- PRs should include screenshots only if the change affects rendered documentation.

## Agent-Specific Instructions
- Prefer minimal, focused edits.
- If adding new sections, keep total document length within 200–400 words.
